## feat-12c — Правка / перенос / отмена через текст (размер M)

### Команды которые надо понимать
- «перенеси встречу с Катей на 16:00» → изменить время (та же дата).
- «перенеси встречу с Катей на завтра в 11» → изменить дату+время.
- «отмени встречу с Катей» / «убери встречу завтра» → cancel + events.delete.
- «продли отпуск до 15 июня» → изменить end_iso (all-day).
- (опционально) «переименуй … в …» → patch title.

### Ключевой архитектурный момент — INTENT перед действием
Сейчас calendar-detector умеет только ИЗВЛЕКАТЬ новые события. Но входящий текст может быть:
- (a) новое событие → создать (текущий поток),
- (b) команда изменить существующее → перенос,
- (c) команда отменить существующее.

**Нужно: distinguish intent ДО создания.** Иначе «перенеси встречу с Катей на 16» создаст
ВТОРУЮ встречу вместо переноса первой — это баг, который надо предотвратить.

Решение: расширить detector (или новый intent-шаг перед ним) — сначала Haiku определяет
intent = create | modify | cancel | none. Промпт получает СПИСОК активных событий
(`listActive` за ±30 дней) как контекст. Если modify/cancel — Haiku возвращает признаки
целевого события (или его описание для поиска) + новые параметры.

### Поток modify/cancel
1. Intent = modify/cancel → найти целевое событие среди активных:
   - fuzzy по названию (Катя → «Встреча онлайн с Катей Гисич») + по времени/дате если указано.
   - Если несколько кандидатов → карточка «какую? [событие 1] [событие 2]».
   - Если не найдено → «не нашёл такой встречи, уточни».
2. Карточка ПОДТВЕРЖДЕНИЯ (в отличие от создания — создание сразу, изменение через подтверждение):
   - modify: «Меняю: «Встреча с Катей» 15:00 → 16:00. [Да] [Отмена]».
   - cancel: «Отменить «Встреча с Катей» 29 мая 15:00? [Да] [Отмена]».
3. При [Да]:
   - modify → `GoogleCalendarClient.patchEventTime` + `repo.patchTime` (audit пишется).
   - cancel → `GoogleCalendarClient.deleteEvent` + `repo.cancel`.
4. callback_data: `cal_edit_yes:<event_id>:<...>` / `cal_edit_no:<event_id>`.

### Состояние «жду подтверждения»
modify/cancel требуют подтверждения — это conversational state. Хранить in-memory
(Map chat_id → pending edit) ИЛИ в БД (таблица pending_calendar_edits). Для одного пользователя
in-memory достаточно (при рестарте — пользователь повторит команду). НЕ усложнять БД ради этого.

### business_invariant
`calendar_change_via_confirmation` — изменения и отмены ВСЕГДА через подтверждение
(создание — сразу, изменение/отмена — через [Да]). Уже зафиксирован в feature_list feat-12c.

## feat-12e — Проверка занятости слота перед созданием (размер M)

### Поток
Перед `createEvent` (в CalendarWorker, для TIMED событий; all-day не проверяем):
1. Запрос к Google: свободен ли [start, end]? → `freebusy.query` (calendar API) ИЛИ
   `events.list({timeMin, timeMax, singleEvents:true})`. **freebusy чище** — вернёт занятые интервалы.
2. **ВАЖНО: проверять РЕАЛЬНЫЙ Google-календарь**, не только нашу таблицу calendar_events.
   В Google есть встречи, которые Атсыхан ставит руками или из других источников — их надо учесть.
   freebusy смотрит весь календарь primary.
3. Если слот свободен → создать как сейчас (молча + карточка «Создал»).
4. Если занят → НЕ создавать. Карточка-вопрос:
   «🕒 На 29 мая 15:00–16:00 уже занято: «<название занявшего>».
    [Поставить всё равно] [Перенести новую — скажи когда] [Перенести ту, что была]».
   - [Поставить всё равно] → создать поверх (overlap разрешён пользователем).
   - [Перенести новую] → бот ждёт новое время для НОВОЙ встречи (переиспользует ввод времени).
   - [Перенести ту, что была] → перенос СУЩЕСТВУЮЩЕЙ встречи (механизм feat-12c!).
5. All-day события (отпуск) пересекаются со всем timed — по ним МАКСИМУМ предупреждение в карточке
   создания, НЕ блокировать.

### Состояние «жду решения по конфликту»
Conversational state (in-memory pending conflict: что хотели создать + что мешает).

### business_invariant
`no_silent_overlap` — бот не ставит встречу в занятый слот молча; при конфликте всегда
возвращает решение пользователю.

### Схема БД
НЕ меняется. Конфликт-чек — рантайм-проверка через Google API. pending-состояния in-memory.

## КАК ТЕСТИРОВАТЬ — ОБЯЗАТЕЛЬНО (требование Атсыхана, повторено дважды за сессию)

> Пользователь: «ты мне сказал работает или не работает классификатор… Мне нужен конечный
> момент: я пишу сообщение, классификатор сработал и перенесло». НЕ объявлять «работает»

### Правило verification для календаря (см. memory/feedback_calendar_verify_real_google.md)
Запись в нашей БД (`calendar_events.status='active'`) НЕ достаточна. ОБЯЗАТЕЛЬНО проверять
что событие реально в Google через `events.get` / `freebusy`. markActive пишет в БД, но
реальное состояние Google — отдельный факт (особенно для patch/delete).

### Автоматический e2e-скрипт (агент пишет и прогоняет ДО «готово»)
Скрипт `scripts/gcal-e2e-*.ts` который прогоняет ПОЛНЫЙ путь и проверяет Google:
1. Записать raw-сообщение (как делает Telegram-адаптер) → classify → detect/intent → результат.
2. Проверить через `GoogleCalendarClient.getEvent(gcal_event_id)` что в Google реально то что ждём.

**feat-12c e2e сценарии:**
- Создать «встреча с Катей завтра в 15» → `events.get` подтверждает 15:00.
- «перенеси встречу с Катей на 16» → `events.get` подтверждает что в Google теперь 16:00 (НЕ создана вторая встреча — `listActive` показывает 1 событие, не 2).
- «отмени встречу с Катей» → `events.get` возвращает status=cancelled/404 в Google.
- Перенос несуществующей → «не нашёл», ничего не создано/изменено.

**feat-12e e2e сценарии:**
- Создать «встреча в 15:00» → создать «звонок в 15:30» (пересечение) → проверить: карточка
  конфликта пришла, второе событие НЕ создано молча (`listActive` = 1 + pending conflict).
- Разрешить [Поставить всё равно] → теперь 2 события в Google.
- Разрешить [Перенести новую на 17] → новое событие в Google на 17:00, без пересечения.
- Слот свободен → создаётся как обычно (без вопроса).
- Конфликт только с all-day (отпуск) → timed-встреча создаётся (предупреждение, не блок).

### Layer 5 — пользователь подтверждает на реальном сообщении
После авто-e2e — пользователь пишет боту реальные команды и подтверждает что видит результат
в своём Google Calendar + карточки в Telegram. ТОЛЬКО после этого — passing.

### Граница автотеста
Доставку Telegram-сообщения «как пользователь» агент не симулирует (бот production, один токен).
Агент прогоняет путь через raw→worker (как recovery) + реальный Google. Финальную доставку
(пользователь пишет → карточка приходит) проверяет Атсыхан.
